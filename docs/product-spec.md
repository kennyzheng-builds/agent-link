# Agent-to-Agent 通信产品方案

## 一句话价值
让两个 OpenClaw Agent 像"老板的小弟"一样直接协作，人类只需"拍板"和"收结果"。

---

## 用户场景

### 场景1：技术问题求助（截图中的案例）
```
Kenny 的 Agent 遇到飞书权限问题 → 生成连接码 "A3F9K2"
Kenny 把 "A3F9K2" 发给朋友
朋友丢给自己的 Agent → 建立通信
两个 Agent 直接对话解决
→ Kenny 收到："已解决，需要你开启 XX 权限"
→ 朋友收到："帮 Kenny 解决了问题"
```

### 场景2：跨团队协作
```
产品经理 A 的 Agent ←→ 工程师 B 的 Agent
讨论需求细节、排期、接口定义
各自向主人同步进展
需要确认时暂停等待老板决策
```

## 核心流程

### 阶段1：接入（一次性设置）
```
主人访问网站 → 了解产品 → 复制接入链接
↓
发给 OpenClaw："/a2a setup https://a2a.openclaw.ai/connect"
↓
Agent 完成绑定（关联 Google 邮箱作为身份标识）
↓
回复："已接入 A2A 网络，你的 ID: kenny@example.com"
```

### 阶段2：发起协作
```
主人对 Agent 说：
- "帮我找人解决飞书权限问题"
- "创建一个协作会话"
- "/a2a new"
↓
Agent 生成：
- 连接码：A3F9K2（6位，10分钟有效）
- 或短链接：https://a2a.openclaw.ai/j/A3F9K2
↓
主人把码/链接发给另一个主人
```

### 阶段3：接受协作
```
对方主人把码/链接丢给 Agent：
- "加入会话 A3F9K2"
- "/a2a join A3F9K2"
- 或直接点击链接
↓
双方 Agent 建立加密通道
通知各自主人："已与 xxx@example.com 建立连接"
```

### 阶段4：Agent 自主对话
```
Agent A ←→ Agent B（WebSocket/RTC 直连）
├─ 交换问题上下文
├─ 尝试解决方案
├─ 请求工具执行（需 sandbox）
└─ 遇到不确定 → 暂停并通知主人
```

### 阶段5：人类介入点
| 触发条件 | Agent 行为 |
|---------|-----------|
| 需要外部授权 | 暂停对话，通知："需要你登录飞书确认" |
| 涉及敏感操作 | 暂停对话，等待主人明确批准 |
| 对话陷入僵局 | 通知："尝试了 X 种方案未解决，建议人工介入" |
| 主人主动查询 | 实时汇报当前进度和已尝试的方案 |

### 阶段6：结束与归档
```
问题解决 → Agent 向双方主人发送总结
↓
主人确认关闭 → 通道关闭
↓
可选：沉淀经验到共享知识库
```

## 信息架构

### 连接码机制
- **格式**：6位字母数字（如 A3F9K2）
- **有效期**：10分钟（可配置）
- **使用次数**：一次性，用完即焚
- **安全**：仅用于发现，实际连接需双方 Agent 握手验证

### 身份体系
- **主人身份**：Google 邮箱（唯一标识）
- **Agent 身份**：由主人身份派生
- **会话身份**：临时生成，会话结束后销毁

### 通信协议
```
[握手阶段]
POST /handshake
{
  "from": "agent-a@example.com",
  "to": "agent-b@example.com",
  "session_id": "uuid",
  "signature": "..."
}

[数据传输]
WebSocket /session/{id}
{
  "type": "message|action|request_human|close",
  "payload": {...},
  "timestamp": "..."
}
```

## 技术要点

### 安全性
1. **端到端加密**：Agent 间通信 TLS + 可选 e2ee
2. **身份验证**：Google OAuth + 签名验证
3. **权限最小化**：默认 sandbox，敏感操作需主人确认
4. **审计日志**：所有操作可追溯

### 隐私性
- 对话内容仅双方 Agent 可见
- 服务器只做信令转发，不存储消息内容
- 主人可选择是否保存历史记录

### 可靠性
- 连接断开后自动重连
- 会话状态持久化
- 超时机制防止僵尸会话
